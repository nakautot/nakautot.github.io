Manual Scripts vs. GitHub Actions with PR Approvals

üî¥ Current Manual Approach - Security Concerns

High-Risk Issues:

  • AWS credentials on developer machines - If a laptop is compromised, production access is exposed
  • No audit trail - Hard to track who deployed what and when
  • No approval gates - Anyone with scripts can deploy to production
  • Human error prone - Easy to deploy wrong branch, skip tests, or use incorrect environment variables
  • No rollback strategy - Manual process to revert bad deployments
  • Inconsistent deployments - Different developers might have different local environments

‚úÖ GitHub Actions + PR Approvals - Best Practices

Security Benefits:

  • Centralized secrets - AWS credentials stored in GitHub Secrets, not on laptops
  • Full audit trail - Every deployment logged with timestamp, user, commit SHA
  • Required approvals - Enforce 1+ reviewers before merging to main
  • Branch protection - Prevent direct pushes to main/master
  • Environment-specific secrets - Separate staging/prod credentials
  • Temporary credentials - Use OIDC/assume role instead of long-lived keys

Operational Benefits:

  • Automated testing - Run tests before deployment
  • Consistent builds - Same environment every time
  • Easy rollbacks - Redeploy previous commit
  • Deployment status - Visual indicators in PRs
  • Notifications - Slack/email on deployment success/failure
  • Staged rollouts - Deploy to staging first, then prod

üìã Recommended Architecture

# .github/workflows/deploy-production.yml
name: Production Deployment

on:
  push:
    branches: [main]
  workflow_dispatch: # Manual trigger option

jobs:
  deploy-frontend:
    runs-on: ubuntu-latest
    environment: production # Requires approval
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
      - run: npm ci && npm run build
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.AWS_ROLE_ARN }}
          aws-region: us-east-1
      - run: ./scripts/deploy-frontend-s3.sh

  deploy-backend:
    needs: deploy-frontend
    environment: production
    # ... similar pattern

Branch Protection Rules:

  • Require PR reviews (2+ approvers for production)
  • Require status checks to pass
  • Require branches to be up to date
  • No direct pushes to main

üéØ Migration Path

  • Phase 1: Add GitHub Actions (parallel to manual)

    Set up workflows for staging environment first. Test thoroughly without touching production.

  • Phase 2: Add environment protection

    Configure GitHub Environments with required reviewers. Set up AWS OIDC (no long-lived credentials).

  • Phase 3: Deprecate manual deployments

    Remove developer AWS production access. Keep scripts for emergency use only.

  • Phase 4: Add advanced features

    Automated rollbacks on health check failures. Canary deployments. Deployment windows/scheduling.
